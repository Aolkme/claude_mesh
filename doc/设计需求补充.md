Meshtastic 网关 设计补充：

首先是 整个框架：

1.一个后台长期运行的程序，在刚启动时，是不会去连接串口的，需要通过去设置才会去连接串口的Meshtastic节点，然后解析与下行proto数据；

2.通过串口连接上节点后，需要对收到的proto数据进行解析，在/home/olkme/桌面/github-project/claude_mesh/protobufs_protobuf-c 下 我已经存放Meshtastic的proto编译文件 与 我私有开发的thingseye的相关文件内容，这一部分可以直接拿来用来解析 proto数据，且不用自己再去定义解析格式；并且对应的需要把 解析出的数据 按字段去存储起来（在Python 里面 是用JSON格式存储的，但是我们目前开发的是C编程，所以需要确定一下具体什么格式比较合适）

3.关于下行数据的部分细节：一帧数据包的发送，各个字段应该是可由选择的API接口，当然了 一般都会有默认值，因为我们的 CLI 和 UI对于解析和发送内核来讲，不需要把各个字段都涉及到，但是内核要做的足够完整！我们的UI操作界面可以不用那么多功能，内核一定是全能的；

4.网关内部也应该维系一个节点数据表，根据节点ID维系 一个节点的各项数据表，包括我们增加的私有配置部分

5.服务启动流程：启动后台服务，然后打开UI界面 或者 CLI，通过命令行连接串口的Meshtastic，这个时候 串口输出proto数据 以及 调试bug信息，所以要对proto数据 和 debug数据单独的分类，然后将输出的信息给分类，例如最开始会输出 网关节点的 各项信息，那就要把这部分信息划入网关节点信息表里，然后我们接收到其他节点信息 我们就把信息存储到 远程节点信息表；

6.整个网络部署：节点设备打开后，会发送自己的信息到Mesh网络中，网关会收到这个信息，然后可以通过操作（这里传的操作就是 通过UI界面 或者 CLI），一键全部入网/针对某个节点发送入网邀请；这里入网邀请 请看私有配置 文件部分，我们发送SetCompanyKeyRequest，然后节点存储 公司公钥 以及节点信息当做管理员；到这里就完成了 入网了

7.整个解析与发送功能一定要针对各个字段 都是完整的，即便这会让工作量 巨大，但是一切都是值得的，只要内核做的完整 后期只维护 应用层 就是很方便的！

8.我们的 CLI 与 UI界面设计 就像 去连接我们的后台服务，通过连接服务，通过命令行  输出当前的串口 接收proto数据 或者 debug数据 或者我们进行一些下发指令的操作，一些简单的 入网要请，以及一些设置 功能，对入网的设备 下发Channel设置等等 让我们可以远程管理设备。

9.我们的远程设置流程 就是 通过入网的形式 远程节点的管理员设置成我们的网关管理员，那么这个时候 网关下发 我们想要通信 的 信道内容，部署私有配置通道 和 私有工作通道。然后我们要控制节点的Meshtastic的官方配置就通过 AdminModule远程就可以实现了
